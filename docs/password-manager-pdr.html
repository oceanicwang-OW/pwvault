私人密码管理器 PDR

本地优先 · 零知识加密 · 跨平台（Windows / macOS / iOS）— v0.4 修订：库文件定稿为容器格式（§4.3）、合并模型改为带 base 快照的三路合并（§6.4）、加密栈统一为 AES-256-GCM + RustCrypto、补全 FFI 类型定义与回收站接口（§12.7）、新增 iOS 同步适配任务 T4.6（T0–T5 共 34 个任务）

1. 产品概述

1.1 一句话定义

一个本地优先（local-first）的私人密码管理器：所有账号密码以强加密形式存储在本地，支持快速搜索、点击显示密码、自动锁定，并通过"加密库文件 + 用户自有网盘"的方式实现跨设备同步，不依赖任何自建服务器。

1.2 目标平台

平台 优先级 说明
Windows 10/11 P0 主力开发机使用场景
macOS 12+ P0 桌面第二平台
iOS 15+ P0 移动端随时查询
Android P1 架构上天然支持，排期靠后
Linux P2 Flutter 支持，按需开启

1.3 核心原则

  1. 零知识：主密码永不落盘、永不上传；没有主密码，任何人（包括拿到库文件的攻击者）都无法解密数据。
  2. 本地优先：断网完全可用；同步是可选增强，不是必需。
  3. 密码永不明文展示：默认掩码显示，点击才显示，且有自动回隐和自动锁定。
  4. 不重复造轮子但不引入黑盒：加密原语用经过审计的成熟库（RustCrypto 系），不自研密码学。

1.4 明确不做（Non-Goals）

  • 不做浏览器插件自动填充（v1 不做，留作远期）
  • 不做团队共享 / 多用户
  • 不自建同步服务器、不做账号体系
  • 不做 TOTP 之外的 2FA 管理（TOTP 本身列为 P1）

2.2 非功能需求

  • 安全：见第 4 章威胁模型与安全架构，这是本产品的最高优先级。
  • 性能：冷启动到解锁界面 < 1.5s；解锁后列表渲染 < 300ms（3000 条以内）；搜索增量过滤 < 50ms。

  • 可靠性：任何写操作原子化（写临时文件 + rename），断电不丢库。

3. 技术选型

3.1 客户端框架：Flutter（结论）

候选对比：

方案 Windows macOS iOS 评价
Flutter 成熟 成熟 一等公民 一套 Dart 代码全覆盖；插件生态完整；UI 完全自绘，三端体验一致
Tauri 2.x 优秀 优秀 已支持但生态新 桌面端体积小，但 iOS 工具链、插件成熟度仍有风险；前端技术栈对纯 C++ 背景没有额外优势
Electron + 原生 iOS 成熟 成熟 需单独写 Swift 两套代码，维护成本翻倍，pass

选择 Flutter 的决定性理由：iOS 是 P0 平台，Flutter 是"一套代码 + iOS 体验过关"的唯一低风险选项。密码学重活不用 Dart 写，而是通过 FFI 调 Rust（RustCrypto，见 3.2），性能与安全都有保证。

3.2 核心技术栈

层 选型 理由
UI Flutter 3.x + Riverpod 状态管理 响应式、可测试
密码学核心 Rust crate core_crypto，经 flutter_rust_bridge FFI 暴露 Argon2id、AES-256-GCM、安全内存清零都在 Rust 侧完成；Dart 侧不接触密钥原文
加密原语库 RustCrypto 系（aes-gcm / argon2 / zeroize） 纯 Rust、持续维护、经社区审计、跨平台一致；sodiumoxide 已停止维护，不采用
本地存储 SQLite（Rust 侧 rusqlite，内存库 + 容器序列化，见 4.3），应用层字段加密而非 SQLCipher 应用层加密让"库文件 = 同步单元"更可控；SQLCipher 在三端的二进制分发反而增加复杂度；DB 归 Rust 侧管理，Dart 不接触密文与 SQL
密码强度 zxcvbn-dart 业界标准估算

3.3 为什么密码学放 Rust 而不是 Dart

  1. Dart 的 GC 托管内存无法可靠清零，明文密钥可能在堆上残留并被 dump；Rust 侧可用 zeroize 精确控制。
  2. Argon2id 是计算密集型，Rust 原生性能远好于 Dart 实现。
  3. Rust 核心可以独立做单元测试与未来安全审计，边界清晰：Dart 只拿到"句柄"，从不持有 KEK/DEK 字节。

4. 安全架构（核心章节）

4.1 威胁模型

威胁 是否防御 手段
库文件泄露（网盘被盗、设备丢失） ✅ Argon2id + AES-256-GCM，离线暴破成本极高
弱主密码被字典攻击 ⚠️ 缓解 创建时 zxcvbn 强制最低强度 + Argon2id 高成本参数
剪贴板嗅探 ✅ 缓解 30s 自动清空；iOS 标记敏感剪贴板
肩窥（shoulder surfing） ✅ 默认掩码、点击显示、自动回隐
离开设备未锁 ✅ 空闲/后台自动锁定
同设备恶意软件 / 键盘记录器 ❌ 不在范围 操作系统已沦陷时任何密码管理器都无法自保，明确声明
内存取证（运行中 dump） ⚠️ 缓解 Rust 侧 zeroize、锁定即清密钥；无法完全防御

4.2 密钥层级（信封加密）

主密码 (用户记忆，永不存储)
   │  Argon2id(password, salt, m=64MB, t=3, p=4)
   ▼
KEK  ── Key Encryption Key（每次解锁时派生，仅存在于 Rust 侧内存）
   │  AES-256-GCM 解密
   ▼
DEK  ── Data Encryption Key（建库时随机生成 32 字节，加密后存于库头）
   │  AES-256-GCM（每条目独立随机 96-bit nonce）
   ▼
条目敏感字段密文（password、notes、TOTP secret）

为什么用信封加密：修改主密码只需用新 KEK 重新包裹 DEK（毫秒级，同时强制重新生成 KDF salt 与包裹 nonce，防止旧 salt 上的破解成果迁移），无需重加密全部条目；信封结构也为未来扩展其他解锁路径（同一 DEK 被多把 KEK 分别包裹）保留了可能性。

4.3 库文件格式

┌─────────────────────────────────────────────┐
│ magic "PWVAULT1"(8B) + header_len(u32 LE)   │
│ Header (明文, JSON)                          │
│   version: 1                                │
│   kdf: { algo: argon2id, salt, m, t, p }    │
│   dek_nonce: base64(12B)                    │
│   wrapped_dek: base64(                      │
│     AESGCM(KEK, DEK, aad=Header其余字段      │
│            的规范化序列化)                    │
│   )                                         │
├─────────────────────────────────────────────┤
│ Body: SQLite 序列化镜像                      │
│   entries 表全部业务字段为 AES-GCM 密文 blob │
├─────────────────────────────────────────────┤
│ Trailer: BLAKE3 校验和（损坏检测，非防篡改）  │
└─────────────────────────────────────────────┘

容器格式（v0.4 定稿）：SQLite 要求自身文件头位于偏移 0，因此库文件不是可直接打开的 SQLite 文件，而是容器。解锁流程：校验 Trailer（不符报 ErrCorrupted，引导从 backups/ 恢复）→ 解析 Header → Argon2id 派生 KEK → 在 AAD 绑定下解包 DEK——Header 任一字段被篡改即解包失败（报 ErrHeaderTampered），天然防 KDF 参数降级攻击 → Body 经 rusqlite deserialize 装入内存 SQLite。保存流程：内存库 serialize → 写 vault.tmp → fsync → 原子 rename，天然满足 2.2 的原子写要求；3000 条 < 15MB 全量驻内存，可接受。

设计权衡（背景记录）：标题/用户名/URL 保持库内明文是为了"未解锁不可见、解锁后零延迟搜索"。但因为库文件会上网盘，v1 决定元数据也加密，搜索改为解锁后在内存中建立索引（见 6.1），代价是解锁时多一次全量解密元数据（3000 条约 100–200ms，可接受）。最终结论：库内所有业务字段全部加密，只有 Header 明文。

4.4 锁定与内存安全

  • 触发锁定：空闲超时（默认 5min，可配 1–30min）/ App 进后台（移动端立即，桌面端可配）/ 系统锁屏事件 / 手动 Cmd+L / Ctrl+L。
  • 锁定动作：Rust 侧 zeroize 清除 KEK、DEK 与所有已解密明文缓存 → Dart 侧清空内存索引与列表数据 → UI 切换到解锁页。
  • 防截屏：iOS 进后台时叠加遮罩快照；Android FLAG_SECURE；桌面端不强制。

4.5 剪贴板策略

  • 复制密码后启动 30s 倒计时（UI 有可视倒计时条），到期写入空串覆盖。
  • iOS：UIPasteboard 设置 localOnly + expirationDate，避免 Universal Clipboard 把密码同步到其他设备。
  • Windows：复制时附带 ExcludeClipboardContentFromMonitorProcessing 与 CanIncludeInClipboardHistory=0 格式标记，阻止密码进入 Win+V 剪贴板历史与跨设备云剪贴板（否则"30s 清空"对历史记录无效）。
  • 倒计时期间若用户再次复制其他内容，则放弃覆盖（避免误删用户新剪贴板）——通过比对剪贴板内容哈希判断。

5. 数据模型

5.1 entries 表

字段 类型 加密 说明
id TEXT (UUIDv7) - 主键，时间有序，利于同步排序
title_ct BLOB ✅ 标题密文
username_ct BLOB ✅ 用户名密文
password_ct BLOB ✅ 密码密文
url_ct BLOB ✅ URL 密文
notes_ct BLOB ✅ 备注密文
totp_ct BLOB ✅ otpauth URI 密文，可空
tags_ct BLOB ✅ JSON 数组密文
favorite INTEGER - 常用标记（0/1），支撑侧边栏"常用"过滤；与时间戳同级的低敏感元数据
created_at / updated_at INTEGER - Unix ms，同步用
deleted_at INTEGER NULL - 软删除墓碑，同步需要
rev TEXT - 修订标识 device_id:counter，冲突检测用

每个密文 blob 格式：nonce(12B) || ciphertext || gcm_tag(16B)，AAD 绑定 entry_id + 字段名，防止密文跨字段/跨条目移植攻击。

5.2 password_history 表（P2）

entry_id, password_ct, replaced_at，每条目保留最近 10 条。

5.3 meta 表

库级配置（自动锁定时长、剪贴板时长、同步配置等），值加密存储。

5.4 库外 app 级配置

最近打开的库列表、窗口尺寸、主题、各条目最近使用时间（搜索排序用）等存放于平台本地配置目录（明文、不参与同步），不得写入任何敏感字段。"最近使用时间"特意放库外：若入库，每次复制密码都会触发一次写库 + 网盘同步，得不偿失。库文件路径本身视为低敏感信息。

6. 关键功能设计

6.1 搜索（F-03）

方案：解锁时构建内存索引，不用 SQLite FTS。

理由：所有字段在库内是密文，FTS 无法直接索引；个人规模（几百到几千条）内存过滤绰绰有余。

流程： 1. 解锁成功 → Rust 侧批量解密全部条目的 title/username/url/tags（不解密 password/notes）返回 EntryMeta 列表 → Dart 侧构建内存索引（拼音处理用 lpinyin，Dart 生态更全，避免在 Rust 重复造拼音轮子）。 2. 索引结构：每条目预计算 searchBlob = lower(title) + username + domain(url) + tags 以及拼音全拼 + 首字母串（用 lpinyin 包，例：淘宝 → taobao / tb）。 3. 输入框 150ms debounce，多关键词 AND 匹配（空格分隔），子串匹配即可命中。 4. 排序：前缀命中 > 标题命中 > 其他字段命中，同分按最近使用时间（读自库外本地配置，见 5.4）。 5. 3000 条 × 平均 100 字符的线性扫描在 1ms 量级，无需更复杂的结构；若未来超 5 万条再引入倒排。

安全注意：索引只含元数据，密码明文从不进入索引；锁定时索引整体销毁。

6.2 密码显示交互（F-04）

  • 列表页：密码列不展示任何掩码（连长度信息都不泄露），仅详情页有密码字段。
  • 详情页：默认显示固定 8 个 •（不反映真实长度）；点击眼睛图标 → 调 Rust 解密该单条密码 → 显示明文 + 启动 10s 回隐倒计时；再次点击立即回隐。
  • 显示明文期间：切后台 / 触发锁定 → 立即回隐并清明文。
  • 解密粒度是单条目按需解密：即使解锁状态下，全库密码明文也不会同时存在于内存。

6.3 密码生成器（F-08）

  • 参数：长度（8–64，默认 20）、大小写/数字/符号开关、排除易混淆字符（0O1lI）开关。
  • 随机源：Rust OsRng（系统 CSPRNG），拒绝取模偏差（rejection sampling）。
  • 附"短语模式"：Diceware 风格 4–6 个单词，便于需要手输的场景。
  • 实时显示 zxcvbn 强度评分与预估破解时间。
  • 局限：zxcvbn 词典以英文为主，对拼音/中文习惯密码会高估强度，评分仅作为最低门槛参考。

6.4 同步（F-10）

方案：单库文件 + 用户网盘 + 条目级三路合并。

  • 用户把库文件放在 iCloud Drive / OneDrive / Dropbox / 任意同步文件夹即可，零服务器成本，符合零知识。
  • 平台差异："任意同步文件夹"仅适用于桌面端。iOS 沙盒无法常驻访问第三方网盘目录，iOS 同步路径仅两条：① iCloud Drive 应用容器（NSFileCoordinator 协调读写 + NSMetadataQuery 监听变更）；② 内置 WebDAV 客户端。对应任务 T4.6。
  • base 快照：每次成功保存/合并后，在本地 sync_base/ 保留一份当前库快照作为下次合并的共同祖先（base），使合并成为真正的三路合并。
  • 难点是冲突：两台设备离线各改了库，网盘只会产生"冲突副本文件"。应对：
  • 应用监测同目录下的冲突副本（如 vault (冲突 1).pwvault）。
  • 检测到后解开两份，以 base 为祖先按 entry.id 做条目级三路合并：
    • 仅一侧相对 base 修改 → 取修改侧；
    • 两侧都改（rev 分叉）→ 取 updated_at 较新者为主，较旧者整条写入冲突区，UI 提示用户人工确认；
    • 墓碑（删除）与修改冲突 → 保留修改，提示用户；
    • base 缺失（首次同步/快照丢失）→ 退化为两路合并 + LWW，所有分歧条目一律进冲突区交人工确认。
  • 主密码不一致：若一侧改过主密码，冲突副本无法用当前密码解开（DEK 的包裹密钥不同），此时报 ErrWrongPassword 并提示用户输入该副本的主密码（即 vault_merge 的 password 参数）；合并写回后整库统一为当前主密码。
  • 合并完成后写回主文件、更新 base 快照、删除冲突副本。
  • 每次写库采用"写 vault.tmp → fsync → 原子 rename"，并在本地 backups/ 目录滚动保留最近 20 个版本，防同步层损坏。
  • WebDAV（坚果云等国内场景）作为 P1 内置客户端实现：基于 ETag 的乐观锁，上传前比对远端 ETag，不一致则先拉取合并。

6.5 导入导出（F-11）

  • 导入：Chrome/Edge/Safari CSV、1Password CSV、Bitwarden JSON；导入过程全内存完成，源文件读取后提示用户安全删除。
  • 导出：① 加密备份（即库文件副本）；② 明文 CSV——需重新输入主密码确认，导出页红色警告。

6.6 TOTP（F-12）

  • 录入：粘贴 otpauth URI 或手输 secret（移动端 P2 加扫码）。
  • 展示：详情页显示 6 位码 + 30s 圆环倒计时，点击复制（同样 30s 清剪贴板）。
  • RFC 6238 实现放 Rust 侧，secret 与密码同等级加密；动态码经 totp_now(h, entry_id) 获取，secret 不出 Rust 层（录入/编辑时的瞬时传入除外）。

7. UI 原型与交互规格

本章为已确认的 UI 原型定稿。可视化高保真原型见配套的 HTML 版文档（password-manager-pdr.html），本节用线框图 + 规格表描述布局约束。

7.1 导航流

        ┌──────────────────────────────────────────────┐
        │                （任意页面触发自动锁定均回到此页） │
        ▼                                              │
   ┌─────────┐   主密码验证通过    ┌──────────┐   选中   ┌──────────┐
   │ 解锁页   │ ────────────────▶ │ 主界面/列表│ ──────▶ │ 详情面板  │
   └─────────┘                   └──────────┘         └──────────┘

锁定触发条件：空闲超时（默认 5min）/ App 进后台（移动端立即）/ 系统锁屏 / 手动 Ctrl/Cmd+L。锁定后清除内存密钥与搜索索引。

7.2 解锁页

规格要点：

  • 无账号输入框：零知识本地架构没有服务端账号体系，"库文件 + 主密码"即登录凭证。
  • 解锁方式仅主密码一条路径（v0.2 决策，已移除生物识别）。
  • 错误反馈：输入框抖动 + 红色描边；连续 5 次错误后强制等待，等待时长按 2^n 秒递增（本机防暴力）。
  • 库选择器下拉展示最近打开的库列表，支持"打开其他库文件…"。

7.3 桌面端主界面（三栏）

规格要点：

  • 三栏宽度：侧边栏 140px 固定、列表栏 220px 固定、详情自适应；窗口 < 900pt 时侧边栏折叠为汉堡菜单，退化为两栏。
  • 列表行 = 色块图标（按标题首字符着色，不联网拉 favicon，避免向第三方泄露所存网站）+ 标题 + 用户名灰字。
  • 搜索框常驻列表栏顶部，输入 150ms debounce 实时过滤，命中拼音首字母时在结果区标注"拼音匹配"。
  • 详情面板字段统一组件：标签 | 值 | 操作按钮组；密码行底色加深以示敏感。
  • 剪贴板倒计时条：复制密码后出现在详情面板底部，进度条 + 剩余秒数，30s 归零自动清空剪贴板。
  • 快捷键：Ctrl/Cmd+F 聚焦搜索、Ctrl/Cmd+L 锁定、Enter 复制选中条目密码、Ctrl/Cmd+N 新建。

7.4 移动端（iOS，单栏导航栈）

规格要点：

  • 标签过滤用顶部横向 chips（单手友好），不用侧边抽屉；新建条目用右下角 FAB。
  • 密码点击显示后，字段标签处直接显示回隐倒计时（"密码 · 7 秒后隐藏"）。
  • 切到后台：立即回隐明文 + 叠加遮罩快照 + 锁定保险库。
  • TOTP 行显示 6 位动态码 + 剩余秒数，点击复制（同样走 30s 剪贴板清理）。

11. 开放问题

  1. 库内是否需要支持多保险库（工作/个人分库）？v1 暂定单库 + 标签即可。
  2. （已决，v0.4）Android 随 M3 顺带启用：Flutter 下成本极低，T3.4 已预留 FLAG_SECURE。
  3. 浏览器填充的远期形态：做插件，还是桌面端全局快捷键 + 自动输入（keyboard emulation）？倾向后者，复杂度低且不碰浏览器扩展权限。

12. 实施级任务拆分（面向 AI 编程）

12.0 AI 编程协作约定

  • 任务粒度：每个任务 = 一次独立的 Claude Code 会话可完成的工作量（约 0.5–1.5 天人力当量），有明确的产出文件与可执行的验收标准。
  • 派发上下文：每次派发任务时附带：本 PDR 相关章节 + 该任务的接口契约 + 上游任务已产出的接口文件。禁止让代理"自由发挥"修改契约；契约变更需回写本文档。
  • 测试先行：Rust 侧任务一律先写测试（含已知向量）；Dart 业务层任务要求单元测试；UI 任务以 widget test + 手工验收清单收尾。
  • 分层禁令：UI 层禁止直接 import bridge 生成代码，必须经 services 层；Dart 任何位置禁止持有 KEK/DEK 字节（只允许 VaultHandle 句柄与单条明文密码的瞬时引用）。
  • 命名：任务 ID 即分支名（task/t1-2-cipher）与提交前缀（[T1.2]）。

                    symbols: bool, exclude_ambiguous: bool }
struct TotpCode   { code: String, remaining_secs: u8 }
struct MergeReport{ merged: u32, conflicts: Vec<EntryMeta> /* 入冲突区的旧侧 */, used_base: bool }
enum   VaultState { Locked, Unlocked, Locking }

// ---- 生命周期 ----
fn vault_create(path: String, password: SecretString, kdf: Option<KdfParams>) -> Result<VaultHandle>;
fn vault_unlock(path: String, password: SecretString) -> Result<VaultHandle>;
fn vault_lock(h: VaultHandle);                                  // zeroize KEK/DEK 与缓存
fn vault_change_password(h: VaultHandle, old: SecretString, new: SecretString) -> Result<()>;  // 强制换 salt + dek_nonce

// ---- 条目 ----
fn entry_list_meta(h: VaultHandle) -> Vec<EntryMeta>;            // 不含已删除；永不含密码
fn entry_list_trash(h: VaultHandle) -> Vec<EntryMeta>;          // 回收站（deleted_at 非空）
fn entry_reveal_password(h: VaultHandle, id: String) -> Result<SecretString>;
fn entry_get_full(h: VaultHandle, id: String) -> Result<EntryPlain>;   // 编辑页用
fn entry_upsert(h: VaultHandle, e: EntryPlain) -> Result<EntryMeta>;
fn entry_soft_delete(h: VaultHandle, id: String) -> Result<()>;
fn entry_restore(h: VaultHandle, id: String) -> Result<()>;
fn entry_purge(h: VaultHandle, id: String) -> Result<()>;       // 回收站彻底删除（保留墓碑供同步）

// ---- 工具 ----
fn generate_password(opts: GenOptions) -> String;                // OsRng + rejection sampling
fn generate_passphrase(words: u8, sep: String) -> String;
fn totp_now(h: VaultHandle, entry_id: String, at_unix: i64) -> Result<TotpCode>;  // secret 不出 Rust

// ---- 同步 ----
// password 仅在冲突副本主密码与当前不同时需要（见 6.4），同密码时传 None
fn vault_merge(h: VaultHandle, conflict_path: String, password: Option<SecretString>) -> Result<MergeReport>;

// Dart 侧分层（UI 只见 services，不见 bridge）
abstract class VaultService {
  Future<void> unlock(String path, String password);
  Future<void> lock();
  Stream<VaultState> get state;            // locked / unlocked / locking
  Future<List<EntryMeta>> listMeta();
  Future<String> revealPassword(String id);  // 调用方负责 10s 内丢弃引用
  Future<EntryMeta> upsert(EntryDraft draft);
}
